# TaskManager 目标文档 v1.1

## 文档定位

这份文档定义 `TaskManager` 项目在 `v1.1` 版本中的目标范围。  
`v1.1` 是当前正在施工的版本，所有关联任务都必须落到本版本文档中的某个阶段。

## 版本定义

- 项目名称：`TaskManager`
- 当前版本：`v1.1`
- 版本主题：建立“项目 -> 版本 -> 文档 -> 阶段 -> 任务”的最小闭环，并把项目总页与任务新增流程接到版本文档上
- 版本原则：本版本先完成结构、总览和文档先行流程，不追求一次性做完全部自动化

## 版本总目标

1. 项目需要支持版本化管理，而不是只保留一个平铺的当前状态。
2. 每个版本都要拥有自己的目标文档和施工文档。
3. 文档中的阶段要能成为任务的归属来源。
4. 当前工程 `TaskManager` 要先成为 `v1.1` 的第一个样板项目。
5. 多项目主页需要直接看见当前版本进度和下一个目标。
6. 新增任务前必须先完成目标文档与施工文档的更新准备。
7. 老项目进入新一轮迭代时，必须能在原项目下新增版本并切换当前版本。
8. 开发中的接口或功能推进，需要能自动接入成任务并持续更新任务状态。

## 阶段定义

### G1 主端与子项目工作台模型

- 主端负责统一管理项目入口。
- 子项目负责承接版本、文档、任务与后续状态推进。
- 项目详情页需要从“项目信息页”升级为“子项目工作台”。

### G2 版本化文档体系

- 一个项目可以包含多个版本。
- 每个版本必须有独立的目标文档与施工文档。
- 当前界面至少要能明确显示当前版本是 `v1.1`。

### G3 文档阶段与任务映射

- 文档内部需要拆出明确阶段。
- 任务必须能关联到：
  - 版本
  - 文档
  - 文档阶段
- 任务完成情况应该反映该阶段的落地进度。

### G4 工程自跟踪样板

- 当前 `TaskManager` 工程本身需要作为一个内置项目出现。
- 当前工程的 `v1.1` 文档、阶段和任务必须可以直接查看。
- 当前工程的迭代进度要通过版本面板展示出来。

### G5 项目总页版本目标透出

- 多项目主页里的每个项目都要显示当前版本的完成率。
- 多项目主页里的每个项目都要显示“当前正在进行或下一个未完成的目标”。
- 项目完成率以当前版本完成率为准，不再采用项目级混合统计。

### G6 任务新增前文档更新规范

- 每次新增任务前，都要先更新当前版本的目标文档和施工文档。
- 这次更新需要留下明确说明，不能只靠口头约定。
- 新增任务时要同时确认任务准备归属到哪个文档阶段。

### G7 老项目新版本迭代

- 已经存在的项目不能只停留在一个版本上。
- 老项目需要在原项目下直接新增新版本，而不是重新建一个新项目。
- 新版本需要自动带上版本文档、阶段模板和当前版本切换。

### G8 开发事件自动接入

- 开发中的接口、模块或功能，需要能通过接入键自动落成任务。
- 第一次接入时要自动创建任务，后续接入要自动推进同一任务状态。
- 这套接入逻辑要能被后续的 OpenClaw 或外部工具复用。

## 本版本验收标准

- 项目详情里能看到当前版本 `v1.1`。
- `v1.1` 目标文档和施工文档能单独展示。
- 文档阶段被明确标识并能对应任务。
- 当前工程的里程碑任务属于 `v1.1`。
- 多项目主页能看到每个项目的当前版本进度和下一个目标。
- 新增任务前必须经过“两份文档更新 + 目标阶段选择”的准备步骤。
- 老项目可以在项目内新增新版本，并把它切成新的版本工作台。
- 开发事件接入后，系统能自动创建任务或推进现有任务。

## 后续版本预留

- 子项目目录状态文件落地
